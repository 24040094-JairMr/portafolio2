Tema 1

Paradigma del Desarrollo de Aplicaciones Orientadas a Servicios

Evolución de las arquitecturas de software hacia modelos basados en servicios independientes y reutilizables

¿Qué es el Desarrollo Orientado a Servicios?

El Desarrollo Orientado a Servicios (DOS) es un paradigma arquitectónico que construye aplicaciones como conjuntos de servicios autónomos, desacoplados y reutilizables. A diferencia del desarrollo monolítico tradicional, DOS permite que cada servicio:

Funcione de manera independiente
Se comunique mediante interfaces estándar
Sea reutilizable en diferentes contextos
Escale de forma individual según demanda

Distinguir los servicios que se ofrecen en la nube

Los servicios en la nube representan la evolución natural del desarrollo orientado a servicios, ofreciendo capacidades de procesamiento, almacenamiento y software a través de Internet.

Modelos principales de servicios cloud:

Modelos Cloud

IaaS - Infraestructura como Servicio

Provee: Recursos de computación básicos (servidores virtuales, almacenamiento, redes)
Ejemplos: Amazon EC2, Microsoft Azure VMs, Google Compute Engine
Control: Usuario gestiona SO, aplicaciones y datos

PaaS - Plataforma como Servicio

Provee: Entorno de desarrollo y despliegue (middleware, bases de datos, herramientas)
Ejemplos: Google App Engine, Heroku, AWS Elastic Beanstalk
Control: Usuario gestiona solo aplicaciones y datos

SaaS - Software como Servicio

Provee: Aplicaciones completas accesibles vía navegador
Ejemplos: Gmail, Salesforce, Office 365, Dropbox
Control: Proveedor gestiona todo, usuario solo usa la aplicación

Características de las Aplicaciones Orientadas a Servicios

Arquitectura SOA

Las aplicaciones construidas bajo el paradigma SOA presentan características distintivas que las diferencian de las aplicaciones tradicionales:

Desacoplamiento

Los servicios son independientes entre sí, lo que permite modificarlos, actualizarlos o reemplazarlos sin afectar a todo el sistema.

Reutilización

Un mismo servicio puede ser utilizado por múltiples aplicaciones, reduciendo la duplicación de código y esfuerzo de desarrollo.

Escalabilidad

Cada servicio puede escalarse de manera independiente según sus necesidades específicas de rendimiento.

Interoperabilidad

Uso de estándares abiertos que permiten la comunicación entre servicios desarrollados en diferentes tecnologías.

Descubribilidad

Los servicios pueden ser descubiertos dinámicamente a través de registros de servicios (como UDDI).

Autonomía

Cada servicio tiene control sobre la lógica que encapsula y puede funcionar de manera independiente.

Aplicaciones Web Híbridas (Mashup)

Concepto de Mashup

Un mashup es una aplicación web híbrida que combina datos, funcionalidades o presentaciones de dos o más fuentes para crear un nuevo servicio.

Los mashups son la manifestación práctica del principio de reutilización en SOA, permitiendo crear aplicaciones innovadoras rápidamente mediante la integración de servicios existentes.

Mashup Diagram

Tipos de Mashups

Mashups de Datos

Combinan datos de múltiples fuentes (ej: Google Maps + datos inmobiliarios)

  • APIs de mapas + bases de datos locales
  • Datos públicos + información privada
  • Feeds RSS + contenidos personalizados
Mashups de Negocios

Integran servicios de diferentes empresas (ej: PayPal + sistema de inventario)

  • Procesamiento de pagos + CRM
  • Logística + tracking de envíos
  • Reservas + sistemas de recomendación
Mashups de Consumidor

Dirigidos a usuarios finales (ej: YouTube + comentarios de Facebook)

  • Redes sociales + multimedia
  • Blogs + sistemas de comentarios
  • Calendarios + recordatorios

Ejemplo Práctico: Mashup de Bienes Raíces

Google Maps API

Portal de Bienes Raíces Interactivo

Este mashup combina tres servicios independientes para crear una aplicación que muestra propiedades en un mapa con información detallada y valores de mercado actualizados.

Arquitectura Orientada a Servicios (SOA)

Marco arquitectónico para diseñar, desarrollar e integrar sistemas empresariales como servicios

¿Qué es la Arquitectura Orientada a Servicios (SOA)?

SOA (Service-Oriented Architecture) es un estilo arquitectónico que estructura una aplicación como una colección de servicios débilmente acoplados que se comunican a través de interfaces bien definidas utilizando protocolos estándar.

Principios de Diseño en SOA

Los principios de diseño SOA guían la creación de servicios que sean verdaderamente orientados a servicios y maximicen los beneficios de esta arquitectura:

01

Contrato de Servicio Estándar

Los servicios se adhieren a un contrato de comunicación estandarizado que define términos, formatos y protocolos.

02

Bajo Acoplamiento

Los servicios mantienen relaciones que minimizan las dependencias y solo requieren conocimiento de cada uno.

03

Abstracción

Los servicios ocultan su lógica interna al mundo exterior, exponiendo solo el contrato de servicio.

04

Reutilización

La lógica del servicio se divide en procesos con la intención de promover su reutilización.

05

Autonomía

Los servicios tienen control sobre la lógica que encapsulan dentro de su límite.

06

Sin Estado

Los servicios minimizan la retención de información específica de una actividad.

07

Descubribilidad

Los servicios se complementan con metadatos comunicativos mediante los cuales pueden ser descubiertos e interpretados.

08

Composición

Los servicios son componentes efectivos de composición, independientemente del tamaño y complejidad de la composición.

Estándares Relacionados con los Servicios

Protocolos y formatos que hacen posible la interoperabilidad en SOA

Importancia de los Estándares en SOA

Los estándares son fundamentales en SOA porque permiten la interoperabilidad entre servicios desarrollados con diferentes tecnologías, por diferentes equipos y en diferentes momentos.

1998

XML (eXtensible Markup Language)

Lenguaje de marcado para estructurar datos de manera legible tanto para humanos como para máquinas.

1999

SOAP (Simple Object Access Protocol)

Protocolo para el intercambio de información estructurada en la implementación de servicios web.

2001

WSDL (Web Services Description Language)

Lenguaje basado en XML para describir interfaces de servicios web.

2000

UDDI (Universal Description, Discovery and Integration)

Registro basado en XML para publicar y descubrir servicios web.

2000

REST (Representational State Transfer)

Estilo arquitectónico que utiliza HTTP para la comunicación entre sistemas.

XML (eXtensible Markup Language)

W3C Recommendation - 1998

Lenguaje de marcado que define un conjunto de reglas para la codificación de documentos en un formato legible tanto para humanos como para máquinas.

Características Clave:

Extensible - Puede crear sus propias etiquetas
Independiente de plataforma y lenguaje
Auto-descriptivo - Las etiquetas describen el contenido
Soporte Unicode - Internacionalización completa

Ejemplo de Documento XML:

<?xml version="1.0" encoding="UTF-8"?>
<pedido id="12345">
    <cliente>
        <nombre>Juan Pérez</nombre>
        <email>juan@example.com</email>
        <direccion>
            <calle>Av. Principal 123</calle>
            <ciudad>Ciudad de México</ciudad>
        </direccion>
    </cliente>
    <productos>
        <producto id="P001" cantidad="2">
            <nombre>Laptop Dell XPS</nombre>
            <precio>1500.00</precio>
        </producto>
    </productos>
    <total>3000.00</total>
</pedido>

SOAP (Simple Object Access Protocol)

W3C Note - 1999

Protocolo de mensajería basado en XML para intercambiar información estructurada en la implementación de servicios web sobre redes.

Estructura de un Mensaje SOAP:

SOAP Envelope
SOAP Header (Opcional)
SOAP Body
SOAP Fault (Opcional)

Ejemplo de Mensaje SOAP:

<?xml version="1.0"?>
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
    <soap:Header>
        <auth:Autenticacion xmlns:auth="http://example.com/auth">
            <auth:Usuario>admin</auth:Usuario>
            <auth:Token>abc123xyz</auth:Token>
        </auth:Autenticacion>
    </soap:Header>
    <soap:Body>
        <ped:ObtenerPedido xmlns:ped="http://example.com/pedidos">
            <ped:IdPedido>12345</ped:IdPedido>
        </ped:ObtenerPedido>
    </soap:Body>
</soap:Envelope>

WSDL (Web Services Description Language)

W3C Note - 2001

Lenguaje basado en XML para describir la funcionalidad ofrecida por un servicio web. Es el "contrato" entre el proveedor y el consumidor del servicio.

Elementos Principales de WSDL:

Types

Definiciones de tipos de datos usando XML Schema

Message

Definición abstracta de los datos transmitidos

PortType

Conjunto abstracto de operaciones soportadas

Binding

Protocolo y formato para un portType específico

Port

Dirección de red para un binding

Service

Colección de endpoints relacionados

UDDI (Universal Description, Discovery and Integration)

OASIS Standard - 2000

Registro basado en XML que funciona como un "directorio telefónico" para servicios web, permitiendo a las empresas publicar y descubrir servicios.

Flujo de Trabajo UDDI:

1
Publicar

Proveedor publica información del servicio en el registro UDDI

2
Buscar

Consumidor busca servicios que cumplan ciertos criterios

3
Descubrir

UDDI devuelve información de contacto y WSDL

4
Invocar

Consumidor invoca el servicio usando la información obtenida

REST (Representational State Transfer)

Tesis Doctoral - 2000

Estilo arquitectónico que utiliza HTTP como protocolo de comunicación y se basa en principios de diseño que hacen a los sistemas web altamente escalables.

Principios REST (Restricciones):

Cliente-Servidor

Separación clara de responsabilidades

Sin Estado (Stateless)

Cada petición contiene toda la información necesaria

Cacheable

Las respuestas deben poder ser cacheadas

Sistema en Capas

Arquitectura en capas con intermediarios

Código bajo Demanda (Opcional)

El servidor puede extender funcionalidad del cliente

Interfaz Uniforme

Identificación de recursos, manipulación a través de representaciones, etc.

Métodos HTTP en REST:

POST Crear recurso
GET Obtener recurso
PUT Actualizar recurso
DELETE Eliminar recurso
PATCH Actualizar parcial
